The flowchart of CTI-BPCS is displayed in Fig. 5.1, including the CTI object construction phase, aggregation phase, recording phase, and verification phase. The first phase constructs standardized CTI objects conforming to the STIX 2.1 data model [8]. The CTI objects are then aggregated into a batch file in the second phase according to the pipeline and directional ring strategy. Hereafter, a participating node hitting the eventual aggregation uploads the data to the blockchain network. Once a suspicion occurs to any specific data, the final phase can help an investigator verify if the data have been tampered or not.
5.2.1 CTI Object Construction
The CTI object construction phase assembles standardized threat intelligence objects conforming to the STIX 2.1 data model. Each CTI object is formally defined as:
CTI =(id,type,object_marking_refs,created,modified,content)
where id is the unique identifier of the CTI object formatted as <type>--UUIDv4; type specifies the SDO category of the object, such as indicator, attack-pattern, or threat-actor; object_marking_refs references the applicable TLP 2.0 [10] marking definition to govern the dissemination scope of the CTI object; created and modified correspond to the creation and last modification timestamps defined in Common Properties; and content denotes the type-specific property set of the corresponding SDO, varying according to the value of type. 
For a single security event, the system produces multiple independent CTI objects, as shown in Eq. (5.1).
ɛ ={〖CTI〗_1,〖CTI〗_2,…,〖CTI〗_n,〖SRO〗_1,…,〖SRO〗_m}
(5.1)
Semantic associations among CTI objects are established through SROs, such as indicates, uses, and attributed-to, enabling intelligence objects to form machine-parseable attack chain contexts. All CTI objects and SROs pertaining to the same event are ultimately packaged within a report object, which serves as the submission unit for blockchain anchoring: 
report =( id,object_marking_refs,created,modified,object_refs)
object_refs ={〖CTI〗_1.id,〖CTI〗_2.id,…,〖SRO〗_1.id,…}
Upon completion of report construction, the system computes its summary digest:  as given in Eq. (5.2).
digest=H(report)

where H(⋅) denotes a secure cryptographic hash function [45], ensuring both integrity and uniqueness of the event reference recorded on the blockchain. If digest≠∅, the report is determined to contain verifiable intelligence content and proceeds to the aggregation phase; otherwise, it is classified as invalid and discarded.

5.2.4 Verification Phase
Once the integrity of a CTI object suffers from a question, an investigator can perform the following to verify and restore the intelligence data.
Step 1. The investigator accesses all report objects stored on the target node S_i according to the incident time. The integrity of report objects during the period serves as the verification target. 
Step 2. The investigator applies t_start and t_end to locate the corresponding block and obtain h(bf). 
Step 3. The investigator captures the file name from the block to request the backup of bf from E. After receiving the backup data, the investigator computes the corresponding hash value and compares it with the one stored in the block. If they are the same, the received batch file is intact to carry on the next step. The opposite means that the file has been tampered. 
Step 4. The investigator employs 〖VPK〗_i to extract 〖report〗_i from 〖VSK〗_i (〖report〗_i)and compares its digest with that kept in S_i. If they are equal, 〖report〗_i is complete. Otherwise, the data of S_i has been tampered. Here, 〖report〗_i could be recovered according to the backup data passing the above examination.
